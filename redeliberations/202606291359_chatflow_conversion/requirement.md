# 再审议需求：Dify Workflow 转 Chatflow 可行性与实施方案探讨

## 任务背景

本项目（糖尿病健康管理平台）在 Dify 平台上配置了以下 4 个工作流（Workflow）和若干 Agent/聊天助手，定义于详细设计文档 v4（已作为初始产出导入）：

1. `diabetes-risk-prediction`（风险预测工作流）— 风险评分与建议生成
2. `life-plan-generator`（生活方案工作流）— 饮食与运动方案生成
3. `health-article-generator`（健康资讯工作流）— 资讯分类与文章生成
4. `punch-analysis`（打卡分析工作流）— 打卡数据分析

此外还有 AI 助手（聊天助手/Agent）和医师对话（聊天助手）等对话型应用，本次不作为转化候选（它们已是对话型）。

## 核心任务

探讨将上述合适的 Dify **Workflow** 转化为 **Chatflow**（对话流）的可能性及实施方案。

## 关键约束（必须严格遵守）

**转化为 Chatflow 必须要有需求上的实质性提升，否则保持为 Workflow。**

"需求上的提升"是指：转化为 Chatflow 后，能够实现 Workflow 模式下无法实现或难以实现的功能价值。例如（仅为示例，不一定正确，需独立论证）：

- 在风险预测 Workflow 中，AI 判断现有表单数据已足够支持高风险结论时，立即中断后续判断过程，提前返回结果（提前终止 / 短路优化）
- 通过多轮对话收集缺失信息后再执行判断
- 根据中间结果动态调整后续流程走向

如果某个 Workflow 转为 Chatflow 后只是形式变化而无实质需求提升，则应明确建议保持为 Workflow。

## 期望产出

针对 4 个工作流逐一分析，产出包含：

1. 每个 Workflow 的现状分析（输入、输出、调用模式、业务场景、现有编排逻辑）
2. 转 Chatflow 的需求提升评估（是否有实质性提升？具体是什么？逐一论证）
3. 转化决策（转 / 不转，及理由）
4. 对于决定转化的工作流：详细的实施方案（Chatflow 编排逻辑、节点设计、与现有 Express 后端的对接变更、API 契约影响、前端交互变更、数据库影响等）
5. 对于决定保留的工作流：保持理由说明
6. 整体影响评估（数据库、API 契约、前端、Dify 配置等方面的变更清单）
