## 问题 1：§9.2.2 引用 §8.5 性能标准时存在模式误置

**严重程度**：轻微

**问题描述**：
§9.2.2 在论证风险预测工作流短路收益有限时，写道「总响应时间在 10s 以内（§8.5 AI 生成类 API 3s 内开始返回）」，引用 §8.5 作为响应时间论证依据。但经核验，§8.5（行 6364）原文为「AI 生成类 API（依赖大模型）在 3s 内开始返回 **SSE 数据流**」，该标准针对的是 streaming 模式的 SSE 流首字返回时间。而风险预测工作流使用 blocking 模式（§6.3.5 callWorkflow，`response_mode: 'blocking'`，行 5520），blocking 模式不产生 SSE 流，「3s 内开始返回 SSE 数据流」标准不适用于 blocking 模式的总响应时间论证。§6.3.5 超时策略表（行 5534）明确区分 blocking（15s 连接 + 15s 读取）与 streaming（10s 连接 + 无限制流式）两种模式，两者性能验收标准不同。

**所在位置**：行 6452，§9.2.2 提升类型 1 论证段落

**改进建议**：
将「（§8.5 AI 生成类 API 3s 内开始返回）」替换为对 blocking 模式实际耗时的直接估算依据，如引用 §6.3.5 blocking 模式 15s 读取超时阈值作为上界、并结合 LLM 建议生成节点的典型耗时（5-8s）进行论证。避免将 streaming 模式的 SSE 首字返回标准 cross-reference 至 blocking 模式的响应时间论证。

## 问题 2：§9.3.2「逐项调整」论证依赖未验证的 LLM 行为假设

**严重程度**：轻微

**问题描述**：
§9.3.2 在论证 life-plan-generator 的「动态调整走向」提升类型时，称「当前 PUT /api/plan/adjust 已支持通过 feedback 文本传达此意图，LLM 可根据 feedback 理解『只改晚餐』的语义」「LLM 在收到『只改晚餐，其他不变』的 feedback 时，应保持其他项不变」，并据此结论「此交互模式已由当前架构支持，Chatflow 的逐项调整仅为粒度优化而非新能力」。

经核验 §5.2.2 系统提示词（行 4657-4681），提示词中未显式指示 LLM 在收到调整反馈时保持未提及的方案项不变——提示词仅要求「方案内容必须个性化」「考虑用户饮食偏好和禁忌」「考虑用户运动偏好和身体限制」，未包含「未提及的方案项应保持原样」的指令。因此「LLM 应保持其他项不变」是对 LLM 行为的未验证假设，而非由当前设计保证的能力。实际上，LLM 可能会在重新生成时改变所有方案项的内容。

此外，§9.3.3 保留理由汇总表（行 6554）从另一角度指出「Chatflow 的逐项调整需改为单行 UPDATE 模型，破坏现有方案组语义与打卡外键关联（punch_in.plan_item_id → life_plans.id）」——这实际上承认了当前架构的「整体重新生成」模型（§3.2.14 行 1912-1916：旧组 is_active=0 + 新组整体写入）与真正的「逐项调整」在数据模型语义上存在本质差异（调整后 punch_in 外键指向的旧方案项变为非活跃）。§9.3.2 将此差异归纳为「仅为粒度优化」与 §9.3.3 将其归纳为「破坏现有方案组语义」之间存在表述张力，可能使读者对当前架构是否真正「已支持」逐项调整产生困惑。

**所在位置**：行 6538，§9.3.2 提升类型 3 论证段落；与行 6554 §9.3.3 保留理由表存在表述张力

**改进建议**：
1. 将「LLM 应保持其他项不变」的断言软化表述为「理论上 LLM 可能根据 feedback 保持未提及项不变，但当前 §5.2.2 系统提示词未显式约束此行为，实际效果取决于 LLM 推理质量，非架构层面保证的能力」。
2. 补充说明 §3.2.14 的整体重新生成模型在数据层面的影响：每次调整均生成新 plan_id 和新方案项 ID，旧方案项 is_active=0 后 punch_in.plan_item_id 外键仍指向旧（非活跃）记录，与 Chatflow 原位 UPDATE 单项的语义不同。
3. 统一 §9.3.2 与 §9.3.3 的表述口径：§9.3.2 侧重用户功能需求层面（用户可通过 feedback 表达调整意图），§9.3.3 侧重数据模型实现层面（整体重生成 vs 原位更新的语义差异），避免前者说「已支持」而后者说「破坏」的读者困惑。

## 问题 3：§9.1.2 判定标尺未显式纳入需求定义的「难以实现」维度

**严重程度**：轻微

**问题描述**：
需求关键约束将「需求上的提升」定义为「转化为 Chatflow 后，能够实现 Workflow 模式下**无法实现或难以实现**的功能价值」（requirement.md 行 22）。§9.1.2 判定规则（行 6403）将其转述为「仅当至少一类提升成立、且**无法通过现有 Workflow 能力**（含分支节点、缺失值估算等已验证能力）实现时，才建议转化」，遗漏了「难以实现」这一维度。

「无法实现」与「难以实现」是两个不同门槛：前者指功能上不可能，后者指虽可实现但实现方式笨拙、脆弱或维护成本高。§9.1.2 将判定标准收窄为仅「无法实现」，理论上可能导致某些 Workflow 能实现但实现方式「难以」维护的场景被判定为「保持 Workflow」。

在实际分析中，各工作流的现有实现均非「难以实现」（已落地运行或可用现有分支节点轻松扩展），因此该框架偏差未影响 4 个工作流的最终结论。但作为判定标尺的完整性，框架文字应与需求定义对齐。

**所在位置**：行 6403，§9.1.2 判定规则段落

**改进建议**：
将判定规则中的「无法通过现有 Workflow 能力实现」修改为「无法通过现有 Workflow 能力实现或难以以可接受的成本实现」，与需求定义的「无法实现或难以实现」双门槛对齐。同时在判定标尺的三类提升论证子问题中，可各补充一个「现有 Workflow 实现方式是否笨拙/脆弱/高维护成本」的考量点，使「难以实现」维度在逐工作流论证中可操作化。

1. **需求响应充分**：4 个工作流逐一分析，每个工作流按三类潜在提升（提前终止/多轮补全/动态调整）独立论证，每类均有明确结论。转化决策严格遵守「无实质需求提升则保持 Workflow」约束，4 个决策均有保留理由汇总表支撑。

3. **工程可操作性**：§9.6 整体影响评估明确告知「4 个工作流均保持 Workflow，数据库/API 契约/前端/Dify 配置均无需变更」，实现者可直接据此决定「维持现状」，无歧义。§9.1.2 判定标尺可复用于未来新增工作流的同类评估。

4. **深度满足后续使用需要**：由于 4 个工作流均决策不转化，无需实施方案（Chatflow 编排逻辑、节点设计等），此为关键约束下的合法逻辑结果。保留理由从评分机制、数据采集、输出契约、数据模型、前端架构等多维度论证，深度充分。

上述 3 个轻微问题均不阻断 §9 的可用性，修复后可进一步提升论证严谨性和框架与需求定义的对齐度。
